# 阶段 0：数据分级与模型调用边界清单

> 本文档的工程权威版本位于 `java-ai-application-lab` 仓库。个人知识库或飞书中的副本仅用于阅读和导入。

本文对应 AI 应用开发学习路线中的阶段 0，目标是把“哪些数据能进入模型、哪些数据需要留在本地、哪些数据需要审批”说清楚。

对 Java 后端来说，AI 应用的数据边界要在第一天建立。后续的 Chat、RAG、Tool Calling、Agent、评测和日志都会反复使用这套规则。

本文用于阶段 0 学习和方案设计；仓库当前已经生效的工程契约以 [`engineering-baseline.md`](../reference/engineering-baseline.md) 为准。

## 1. 为什么先做数据分级

LLM 应用会把用户输入、业务上下文、历史对话、检索资料、工具返回结果和系统提示词一起组织成 Prompt。只要这些内容被发送给外部模型服务，就已经发生了数据出域。

阶段 0 先做数据分级，主要解决四个问题：

- 哪些数据可以进入外部模型。
- 哪些数据只能进入私有化模型或本地模型。
- 哪些数据需要脱敏后才能使用。
- 哪些数据需要排除在 Prompt、日志、评测集和截图之外。

这个边界先定下来，后面做功能时就能直接套用，不需要每次临时判断。

## 2. 数据分级建议

学习项目可以先采用 5 级分类，足够覆盖大多数 AI 应用场景。

| 等级 | 名称 | 示例 | 默认处理方式 |
|---|---|---|---|
| L0 | 公开数据 | 官方文档、公开产品说明、公开博客、开源代码片段 | 可进入外部模型 |
| L1 | 内部普通数据 | 内部规范、非敏感技术笔记、通用需求模板 | 脱敏或确认后可进入外部模型 |
| L2 | 内部业务数据 | 客服记录、订单状态、工单内容、内部知识库 | 默认只进私有环境；外部模型需脱敏、审批和最小化 |
| L3 | 敏感数据 | 身份证号、手机号、邮箱、地址、合同、财务、简历、客户信息 | 默认使用私有化或本地模型；结构化字段经可验证脱敏和审批后再评估外发 |
| L4 | 高敏和受控数据 | 密钥、令牌、密码、支付信息、生产数据、权限配置、医疗和法律高风险材料 | 不进入 Prompt、日志、评测集和外部模型 |

学习阶段可以把边界定得更保守：

- L0 可以直接用于外部模型。
- L1 先确认再使用。
- L2 先做最小化、脱敏和审批，再判断是否必须外发。
- L3 优先使用模拟数据、脱敏样本、私有化模型或本地模型。
- L4 只保留在本地或受控系统中。

脱敏只是降低风险的处理，外发仍需要明确授权依据。手机号、身份证号、银行卡号这类结构化字段可以通过规则做较稳定的遮蔽；合同、简历、客服记录、聊天记录这类自由文本更容易因为上下文、稀有组合和行文特征被再识别，默认走私有化或本地模型。

## 3. 模型调用边界

不同模型部署方式对应不同的数据边界。

| 模型方式 | 适合数据 | 说明 |
|---|---|---|
| 外部模型 API | L0、经确认的 L1、经审批且脱敏后的低风险 L2 | 学习和原型阶段最常见 |
| 企业合规模型服务 | L0-L2、经审批且可验证脱敏后的结构化 L3 字段 | 需要确认供应商、租户隔离、日志策略和数据保留策略 |
| 私有化模型服务 | L0-L3 | 适合内部知识库、客服、合同、简历等敏感业务 |
| 本地模型 | L0-L4 中允许本机处理的数据 | 适合实验、敏感数据验证和离线任务 |

实践中需要同时确认模型能力和数据处理边界：

- 输入数据会发到哪里。
- 服务商是否保存请求和响应。
- 是否会用于训练或改进模型。
- 日志保留多久。
- 数据是否跨境。
- 请求链路中还有哪些代理、网关、插件或第三方服务。

这些问题无法确认时，按更高风险处理。

低风险 L2 指经过业务负责人确认、完成字段最小化、脱敏可验证、外发目的明确、无客户隐私原文、无合同或财务明细的数据片段。只要包含长篇自由文本、真实客户上下文或不可验证脱敏内容，就按高风险处理。

## 4. Prompt 数据边界

Prompt 是最容易泄露数据的位置，因为它通常把多种数据拼在一起。

Prompt 中常见数据包括：

- 系统提示词。
- 用户问题。
- 历史对话。
- RAG 检索片段。
- Tool Calling 的工具返回结果。
- 输出格式和示例。

建议规则：

| 数据类型 | 是否进入 Prompt | 处理方式 |
|---|---|---|
| 公开说明和公开文档 | 可以 | 直接使用 |
| 内部普通规范 | 视情况 | 删除组织、项目、人员等敏感标识 |
| 用户输入 | 谨慎 | 作为不可信数据处理，过滤只用于降噪和拦截明显敏感内容 |
| 历史对话 | 谨慎 | 截断、摘要、脱敏后再使用 |
| RAG 检索结果 | 谨慎 | 检索阶段先做权限过滤，并把片段当作不可信资料 |
| 工具返回结果 | 谨慎 | 只返回回答所需字段，并把结果当作不可信资料 |
| 密钥、令牌、密码 | 不进入 | 使用环境变量或密钥管理系统 |
| 真实生产数据 | 默认不进入 | 使用脱敏样本或私有模型 |

Prompt 设计时遵循数据最小化原则：模型只拿完成当前任务所需的最少信息。

Prompt Injection 防护需要后端边界，关键词过滤只是辅助。所有来自外部的文本，包括用户输入、网页内容、RAG 文档片段、工具返回结果和历史消息，都只作为“数据”使用；系统指令和权限来源由服务端定义。真正的安全边界要放在后端：权限校验、工具白名单、参数校验、人工确认和最终业务决策都由后端控制。

## 5. 日志与审计边界

AI 应用的日志很容易把 Prompt、模型响应、工具结果和用户输入一起记录下来。日志系统通常有更多人能访问，风险不低于模型调用本身。

建议日志记录这些信息：

- traceId / requestId。
- 用户 id 或租户 id 的脱敏标识。
- 会话 id。
- 模型名称。
- Prompt 版本。
- 输入 token、输出 token、总 token。
- 请求耗时。
- 检索命中文档 id 和 chunk id。
- 工具名称和调用结果状态。
- 错误码和异常类型。

日志中控制这些内容：

- 不记录 API Key、token、密码、cookie。
- 不记录完整 Prompt，除非进入受控调试环境。
- 不记录完整用户隐私和业务原文。
- 不记录完整工具返回结果。
- 对手机号、邮箱、身份证号、地址、银行卡等做脱敏。
- 高风险操作记录审计事件，但审计日志也要脱敏。
- 对手机号、身份证号、租户 id、用户 id 等低熵标识做伪名化时，优先使用随机化映射表或带保密盐的哈希；直接无盐哈希、短哈希或顺序编号仍可能被枚举反推，或泄露规模和排序信息。

一个比较稳妥的日志格式：

```json
{
  "traceId": "trace-001",
  "tenantRef": "tenant_ref_k7p4x2",
  "userRef": "user_ref_r9m2q6",
  "scene": "rag_qa",
  "model": "xxx-chat",
  "promptVersion": "rag-system-v1",
  "inputTokens": 1200,
  "outputTokens": 280,
  "latencyMs": 2300,
  "retrievalChunkIds": ["chunk-001", "chunk-018"],
  "toolCalls": [],
  "status": "success"
}
```

## 6. RAG 和 Embedding 数据边界

RAG 场景要同时关注原始文档、切块文本、向量、检索结果和最终 Prompt。

RAG 数据链路：

```text
原始文件
-> 文档解析文本
-> chunk 切块
-> embedding 向量
-> 向量数据库
-> 检索结果
-> Prompt 上下文
-> 模型回答
```

每一步都要考虑权限和数据边界：

| 环节 | 风险 | 建议 |
|---|---|---|
| 原始文件 | 包含合同、客户、财务、简历等敏感数据 | 上传前分级，存储时记录权限 |
| 解析文本 | 从 PDF/Word 中提取出完整原文 | 进入受控存储，避免写入普通日志 |
| chunk 切块 | 可能把敏感字段切入片段 | 切块前或切块后做脱敏策略 |
| embedding | 向量可能间接表达原文语义 | 向量库按租户和权限隔离 |
| 检索 | 可能召回用户无权查看的片段 | 检索条件中加入权限过滤 |
| Prompt | 召回片段会发送给模型 | 发送前做二次过滤和脱敏 |
| 回答 | 可能复述敏感信息 | 输出前做安全检查和引用核对 |

Embedding 模型也属于模型调用。把文档送去外部 embedding API，同样是数据出域。内部文档和敏感文档优先考虑企业合规模型、私有化 embedding 或本地 embedding。

## 7. Tool Calling 数据边界

Tool Calling 会把模型和真实业务系统连接起来，风险高于普通聊天。

工具调用中要坚持两条边界：

- 模型只负责选择工具和生成候选参数。
- 后端负责鉴权、校验、执行、审计和最终决策。

工具入参规则：

| 字段来源 | 是否可信 | 处理方式 |
|---|---|---|
| 登录态用户 id | 可信 | 从后端上下文获取 |
| 租户 id / 组织 id | 可信 | 从后端上下文获取 |
| 模型生成的订单号 | 不完全可信 | 做格式校验和归属校验 |
| 模型生成的金额 | 不可信 | 需要业务规则和人工确认 |
| 模型生成的权限字段 | 不可信 | 后端忽略，由权限系统决定 |
| 用户自然语言输入 | 不完全可信 | 解析后仍需校验 |

工具返回结果规则：

- 查询类工具只返回回答所需字段。
- 写操作工具返回执行状态、业务单号和必要提示。
- 敏感字段先脱敏再进入模型上下文。
- 资金、合同、权限、生产操作等高风险工具必须有二次确认。

## 8. Agent 数据边界

Agent 会多步骤执行任务，数据会在步骤之间流转。相比单次聊天，Agent 更需要状态和审计。

Agent 需要记录：

- 任务 id。
- 当前状态。
- 每一步输入摘要、引用 id 或脱敏后的必要字段。
- 每一步输出摘要、结构化状态或脱敏后的必要字段。
- 使用的模型。
- 调用的工具。
- 用户确认记录。
- 失败原因和重试次数。

Agent 需要控制：

- 最大步骤数。
- 最大 token 预算。
- 最大耗时。
- 可调用工具白名单。
- 高风险步骤的人机确认点。
- 每一步可访问的数据范围。

Agent 的中间结果也要分级。模型生成的计划、摘要、候选 SQL、工具参数、风险判断进入后端校验或人工确认后，再转成事实记录或最终业务指令。

Agent trace 同样适用日志与审计边界。普通 trace 记录状态、摘要、引用 id、工具名、耗时和错误码；完整 Prompt、检索原文、工具原始返回和用户隐私只进入受控调试存储，并设置访问权限、保留期限和脱敏策略。

## 9. 评测数据边界

评测集会被反复使用，也可能被提交到仓库、分享给团队或导入评测平台，因此要单独管理。

建议评测集只使用：

- 公开数据。
- 人工编造的模拟数据。
- 脱敏后的结构化业务样本。
- 经批准可用于评测的数据。

评测集如果会提交到仓库、导入外部平台或分享给团队，合同、简历、客服记录、聊天记录等自由文本样本优先使用模拟数据替代。只有结构化、可验证脱敏、审批通过的数据字段才进入可分发评测集。

评测集中避免出现：

- 真实客户信息。
- 真实订单号。
- 真实手机号、邮箱、地址、身份证号。
- 真实合同和财务数据。
- 真实生产问题单。
- API Key、cookie、访问令牌。

如果要验证敏感场景，可以用结构相似的模拟数据替代真实数据。例如把真实订单号替换为 `ORDER-2026-0001`，把手机号替换为 `138****0000`。

## 10. 最小可执行规则

学习阶段先执行这 10 条规则：

1. API Key 只通过环境变量或密钥管理系统使用。
2. `.env` 只留在本地，不提交到版本库。
3. Prompt 中不放密钥、密码、cookie、token。
4. 日志中不打印完整 Prompt 和完整模型响应。
5. 用户隐私进入模型前先脱敏。
6. RAG 检索先做权限过滤，再拼 Prompt。
7. Tool Calling 的用户身份从后端登录态获取。
8. 写操作工具必须二次确认。
9. 评测集优先使用模拟数据。
10. 无法判断数据等级时，按更高风险等级处理。

| 问题 | 检查结果 |
|---|---|
| 这个功能会把哪些数据发给模型？ |  |
| 数据属于 L0-L4 哪一级？ |  |
| 是否调用外部模型 API？ |  |
| 是否需要脱敏？ |  |
| Prompt 中是否包含用户隐私或业务原文？ |  |
| 日志是否会记录 Prompt、响应或工具结果？ |  |
| RAG 检索是否做了权限过滤？ |  |

## 12. 典型场景判断

| 场景 | 推荐处理 |
|---|---|
| 让外部模型解释公开技术文档 | 可以直接使用 |
| 让模型总结内部技术规范 | 先删除内部系统名、人员、链接和敏感配置 |
| 让模型分析真实简历 | 使用脱敏文本或私有模型 |
| 让模型回答客户订单问题 | Tool Calling 查询后只返回必要字段 |
| 让模型读取合同原文 | 默认使用私有模型或本地模型 |
| 让模型生成 SQL | 只生成候选 SQL，后端校验后人工确认 |
| 让模型调用取消订单接口 | 必须确认用户身份、订单归属和二次确认 |
| 把用户对话用于评测 | 优先使用模拟对话；可分发评测集只放结构化、可验证脱敏、审批通过的字段 |
| 把生产错误日志发给模型排查 | 先删除密钥、token、用户信息和内部地址 |

## 13. 和后续阶段的关系

这份清单会在后续阶段持续复用：

- 阶段 1：决定聊天接口能记录哪些日志，历史对话如何脱敏。
- 阶段 2：决定结构化输出结果能否直接进入业务流程。
- 阶段 3：决定 RAG 文档如何入库、检索和拼 Prompt。
- 阶段 4：决定 Tool Calling 能调用哪些工具，哪些操作需要确认。
- 阶段 5：决定 Agent 每一步能访问哪些数据。
- 阶段 6：决定评测集、安全测试和可观测性指标如何设计。
- 阶段 7：决定缓存、模型路由、灰度和成本统计能保存哪些内容。

阶段 0 的目标是先形成可执行的工程边界。只要每个 AI 功能都能回答“数据从哪里来、发到哪里去、谁能看到、保存多久、如何追踪”，后续系统就更容易安全地演进。
