# 阶段 0：必须理解的 AI 应用基础概念

> 本文档的工程权威版本位于 `java-ai-application-lab` 仓库。个人知识库或飞书中的副本仅用于阅读和导入。

本文对应学习路线中的 `5.2 必须理解的概念`，面向有 Java 后端经验、准备进入 AI 应用开发的学习者。

阶段 0 的重点是先建立基础认知：LLM 应用由哪些部分组成，请求模型时发生了什么，成本和效果受哪些因素影响，哪些数据可以进入模型上下文，以及后端系统需要承担哪些工程职责。

## 1. LLM

LLM 是 Large Language Model，大语言模型。它接收一段上下文，然后根据上下文生成后续内容。

在 AI 应用开发里，可以把 LLM 理解成一个“根据输入文本生成输出文本的推理服务”。你给它问题、背景资料、角色要求、输出格式要求，它根据这些内容生成回答。

后端开发时需要重点理解：

- LLM 的输入通常是文本消息，比如 `system`、`user`、`assistant` 多轮消息。
- LLM 的输出也是文本，或者结构化文本，比如 JSON。
- LLM 的回答具有概率性，同样的问题在不同参数下可能得到不同结果。
- LLM 需要通过 Prompt、RAG 或 Tool Calling 获得业务上下文。
- LLM 外部 API 调用会涉及网络延迟、费用、限流、异常、重试和数据安全问题。

Java 后端视角：

```text
用户请求
 -> 后端接口
 -> 组织 Prompt / 上下文
 -> 调用模型 API
 -> 解析模型响应
 -> 返回给前端或进入业务流程
```

阶段 0 要理解到：LLM 是一个生成式推理能力，需要通过上下文和工程约束让它服务于业务目标。

## 2. Token

Token 是模型处理文本时使用的基本计量单位。它可以近似理解为“被模型切分后的文本片段”。

Token 会影响三个核心问题：

- 上下文长度：一次请求能放多少内容。
- 调用费用：多数模型按输入 token 和输出 token 计费。
- 响应延迟：输入越长、输出越长，整体耗时通常越高。

例如一段请求里可能包含：

```text
system 指令
+ 用户问题
+ 历史对话
+ 检索到的知识库资料
+ 输出格式要求
```

这些内容都会消耗输入 token。模型生成的回答会消耗输出 token。

对中文场景要额外注意：

- token 与字符数、词数没有一一对应关系，具体切分方式取决于模型的 tokenizer。
- 中文文本粗略估算时，经常会出现一个汉字对应约 1 个或多个 token 的情况，实际值要以模型 tokenizer 或接口返回的 usage 为准。
- 很多模型的输出 token 单价高于输入 token，长回答、长 JSON、长报告会明显拉高成本。

后端开发时需要重点关注：

- 控制历史对话长度。
- 控制 RAG 检索片段数量。
- 控制输出长度。
- 记录每次调用的输入 token、输出 token 和总 token。
- 基于真实 usage 数据估算成本和性能，字数只作为粗略参考。

阶段 0 要理解到：token 是 AI 应用里的成本、性能和上下文容量单位。

## 3. Context Window

Context Window 是模型一次请求能处理的最大上下文范围。它决定了一次调用中输入和输出合计能占用多少 token。

上下文通常包括：

- 系统指令。
- 当前用户问题。
- 多轮历史对话。
- RAG 检索结果。
- 工具调用结果。
- 输出格式约束。
- 示例数据。

Context Window 规划时要同时考虑输入和输出：

- 输入 token 包括 Prompt、用户问题、历史消息、检索资料和工具结果。
- 输出 token 是模型生成的回答内容。
- 输入内容占用过多时，可用输出空间会变小，长回答可能被截断或触发超限错误。
- 部分模型还会额外设置最大输出 token 上限，需要同时看上下文窗口和输出上限。

Context Window 越大，能放的资料越多，但资料过多会带来几个问题：

- 成本增加。
- 延迟增加。
- 关键信息被无关信息稀释。
- 敏感数据进入上下文的风险增加。

后端开发时常见处理方式：

- 对历史消息做截断或摘要。
- RAG 只放最相关的片段。
- 对资料做分块、召回、重排。
- 输入模型前做脱敏和数据最小化。
- 按业务场景预留输出 token，比如问答、摘要、JSON 抽取、报告生成分别设置不同预算。

阶段 0 要理解到：Context Window 是模型请求的容量边界，AI 应用需要主动管理上下文，并为输出预留空间。

## 4. Prompt

Prompt 是给模型的指令和上下文组织方式。它决定模型“以什么角色、基于什么资料、按照什么规则、输出什么格式”。

一个常见 Prompt 结构包括：

```text
角色：你是一个企业知识库助手
任务：根据给定资料回答用户问题
约束：只基于资料回答，缺少资料时说明无法判断
上下文：检索到的文档片段
输出格式：JSON 或 Markdown
用户问题：xxx
```

Prompt 设计的重点包括：

- 明确任务目标。
- 明确输入资料。
- 明确输出格式。
- 明确边界条件。
- 提供必要示例。
- 控制语气和风格。
- 为异常情况设计响应方式。

Java 后端视角：

- Prompt 适合放在独立文件或配置中，便于版本管理和评审。
- 动态变量由后端填充，比如用户问题、资料片段、语言、输出格式。
- 重要 Prompt 需要版本管理和测试集验证。
- Prompt 变更会影响业务输出，应当像接口规则一样管理。

阶段 0 要理解到：Prompt 是 AI 应用里的“任务说明书”，它直接影响输出质量和稳定性。

## 5. Temperature

Temperature 是控制模型输出随机性的参数。它影响回答的稳定性和创造性。

可以这样理解：

- 较低 temperature：输出更稳定，更适合事实问答、分类、抽取、结构化输出。
- 较高 temperature：输出更发散，更适合创意写作、头脑风暴、营销文案。

常见场景建议：

| 场景 | Temperature 倾向 |
|---|---|
| JSON 信息抽取 | 低 |
| 客服问答 | 低到中 |
| 知识库问答 | 低 |
| 代码生成 | 低到中 |
| 创意文案 | 中到高 |
| 头脑风暴 | 中到高 |

后端开发时需要注意：

- 生产环境的业务流程通常追求稳定性。
- 结构化输出场景通常使用较低 temperature。
- 同一业务场景应固定参数，便于评测和排查问题。
- 参数变化需要记录，方便对比效果。

阶段 0 要理解到：temperature 是输出稳定性和创造性的调节器。

## 6. Hallucination

Hallucination 通常翻译为“幻觉”，指模型生成了看起来合理但事实不准确的内容。

它可能出现在这些场景：

- 模型缺少必要资料。
- 用户问题超出上下文范围。
- Prompt 没有明确回答边界。
- 检索资料质量差。
- 模型把相似概念混在一起。
- 输出格式看似完整，但字段内容并无依据。

业务风险包括：

- 客服给出错误政策。
- 知识库问答编造文档内容。
- 数据分析生成不存在的结论。
- 代码助手引用不存在的 API。
- 自动化流程基于错误信息执行后续动作。

常见控制方式：

- 使用 RAG 提供可信资料。
- 要求模型引用资料来源。
- 缺少依据时返回“无法判断”。
- 对关键字段做程序校验。
- 对高风险操作加入人工确认。
- 对输出做评测和回归测试。

阶段 0 要理解到：幻觉是 LLM 应用的基本风险，需要通过上下文、校验、评测和业务流程控制。

## 7. Embedding

Embedding 是把文本转换成向量的过程。向量可以理解为一组数字，用来表示文本语义。

例如：

```text
“如何重置密码？”
“忘记密码怎么办？”
```

这两句话字面不同，但语义接近。通过 Embedding 转成向量后，它们在向量空间里的距离会比较近。

Embedding 常用于：

- 语义搜索。
- 知识库检索。
- 相似问题匹配。
- 文档去重。
- 推荐系统。
- 聚类分析。

典型流程：

```text
文档文本
 -> 切分成片段
 -> 调用 Embedding 模型生成向量
 -> 存入向量数据库
 -> 用户提问时生成问题向量
 -> 检索最相似的文档片段
```

Java 后端视角：

- Embedding 通常用于检索阶段。
- 向量需要和原始文本、文档 ID、标题、来源、权限信息一起存储。
- 问题向量和库内文档向量必须使用同一个 embedding 模型，并且向量维度要一致。
- 更换 embedding 模型后，旧索引通常需要全量重建，否则相似度检索结果会失真。
- 向量数据库字段维度要和 embedding 模型输出维度一致，比如 `1536`、`3072`、`1024` 等。
- 企业场景要考虑数据权限，用户只能检索自己有权限看的内容。
- 中文场景需要选择中文效果较好的 embedding 模型。

阶段 0 要理解到：Embedding 解决的是“语义相似度检索”问题，是 RAG 的基础能力。

## 8. RAG

RAG 是 Retrieval-Augmented Generation，检索增强生成。它的核心思路是：先从知识库检索相关资料，再让模型基于资料回答。

基本流程：

```text
用户问题
 -> 问题向量化
 -> 向量数据库检索相关片段
 -> 可选 rerank 重排
 -> 拼接 Prompt
 -> 调用 LLM
 -> 返回答案和引用来源
```

RAG 适合解决的问题：

- 企业内部知识库问答。
- 产品文档问答。
- 政策制度问答。
- 客服知识库。
- 技术文档助手。
- 法规、合同、手册类资料查询。

RAG 的关键质量点：

- 文档切分是否合理。
- 检索结果是否相关。
- 排序是否准确。
- Prompt 是否要求基于资料回答。
- 回答是否带来源。
- 权限控制是否生效。
- 资料更新后索引是否同步。

Java 后端视角：

- 文档上传、解析、切分、向量化、入库通常是后端任务。
- 问答接口需要完成检索、组装上下文、调用模型、返回答案。
- 生产场景要记录检索命中的文档、片段、分数和模型回答。
- 对企业数据要做租户隔离、权限过滤和脱敏处理。

阶段 0 要理解到：RAG 是把业务知识接入 LLM 的主流方式。

## 9. Tool Calling

Tool Calling 是让模型根据用户意图选择调用后端函数。模型负责判断“需要调用哪个工具、参数是什么”，后端负责真正执行。

例如用户说：

```text
帮我查一下订单 12345 的物流状态
```

模型可以生成工具调用意图：

```json
{
  "tool": "query_order_shipping",
  "arguments": {
    "orderId": "12345"
  }
}
```

后端接收到后，执行真实业务方法：

```text
queryOrderShipping("12345")
```

然后把结果再交给模型组织成自然语言回答。

Tool Calling 适合：

- 查询订单。
- 查询库存。
- 创建工单。
- 查询账户信息。
- 调用审批接口。
- 执行业务规则校验。
- 连接企业已有系统。

后端开发时必须关注：

- 工具参数校验。
- 用户权限校验。
- 幂等控制。
- 审计日志。
- 错误处理。
- 高风险操作确认。
- 工具返回结果脱敏。

阶段 0 要理解到：Tool Calling 是模型和后端业务系统之间的桥梁，真正的业务执行仍然由后端控制。

## 10. Agent

Agent 是一个有状态、多步骤、可调用工具的任务执行系统。它通常会根据目标拆解任务，调用模型、工具、检索系统，并根据中间结果继续下一步。

一个简单 Agent 流程可能是：

```text
用户目标：帮我分析这个需求并生成任务拆分

1. 理解用户目标
2. 检索相关历史需求
3. 调用工具读取模板
4. 生成初版任务拆分
5. 检查是否缺少字段
6. 补充风险和验收标准
7. 输出最终结果
```

Agent 和普通 Chat 的区别在于：

| 能力 | 普通 Chat | Agent |
|---|---|---|
| 单轮问答 | 支持 | 支持 |
| 多步骤任务 | 较弱 | 较强 |
| 状态管理 | 简单 | 更重要 |
| 工具调用 | 可选 | 常见 |
| 中间过程 | 较少 | 较多 |
| 失败恢复 | 简单 | 需要设计 |

Java 后端视角：

- Agent 需要后端管理状态、步骤、工具、权限和异常。
- Agent 需要明确最大步数、超时时间、失败策略和人工确认点。
- 高风险动作应由业务系统校验和执行。
- Agent 的每一步最好可追踪、可审计、可回放。

阶段 0 要理解到：Agent 是把 LLM、RAG、Tool Calling、状态管理组合起来的任务系统。

## 11. 这些概念之间的关系

可以按下面这条主线理解：

```text
LLM 是核心生成能力
Token 和 Context Window 决定请求容量、成本和延迟
Prompt 决定任务表达方式
Temperature 影响输出稳定性
Hallucination 是需要控制的核心风险
Embedding 负责语义表示
RAG 用 Embedding 找资料，再让 LLM 基于资料回答
Tool Calling 让 LLM 调用后端业务能力
Agent 把 LLM、RAG、Tool Calling 和状态管理组合成多步骤任务系统
```

Java 后端可以把 AI 应用拆成几层：

```text
接口层：接收用户请求，处理鉴权、参数校验、响应格式
编排层：组织 Prompt、上下文、模型参数、工具调用和状态
模型层：调用 LLM、Embedding、Rerank 等模型服务
数据层：存储业务数据、文档、向量、会话、日志和审计记录
治理层：处理成本、限流、观测、安全、评测和回归验证
```

## 12. 阶段 0 的掌握标准

学完 `5.2` 后，至少要能回答这些问题：

1. LLM 应用的一次请求包含哪些内容？
2. token 为什么会影响成本和性能？
3. 为什么中文 token 估算只能把字数作为粗略参考？
4. context window 为什么要同时考虑输入和输出？
5. Prompt 应该包含哪些部分？
6. temperature 在什么场景应该调低？
7. 模型幻觉会带来哪些业务风险？
8. Embedding 为什么能做语义搜索？
9. 更换 embedding 模型为什么通常要重建索引？
10. RAG 的完整流程是什么？
11. Tool Calling 中模型和后端分别负责什么？
12. Agent 和普通聊天接口有什么区别？

这一节的核心目的：先把 AI 应用开发的基础语言统一起来。后面学习 Prompt、结构化输出、RAG、Tool Calling、Agent 时，都会反复用到这些概念。
